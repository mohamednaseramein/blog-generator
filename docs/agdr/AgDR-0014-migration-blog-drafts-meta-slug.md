# Add meta_description and suggested_slug to blog_drafts

> In the context of the Publish step MVP, facing the need to surface AI-generated SEO metadata alongside the blog draft, I decided to execute a schema migration adding two nullable columns to `blog_drafts` to achieve copy-ready meta description and slug fields, accepting a one-time schema change with no downtime.

**Migration type**: schema
**Affected tables / entities**: blog_drafts
**Estimated downtime**: none — both columns are nullable with no backfill required; Postgres adds them instantly
**Data volume**: unknown (existing rows unaffected; columns default to NULL)
**Target environment(s)**: staging → prod

## Context

The Publish step (feature #38) requires surfacing a generated meta description and suggested slug so users can copy them alongside their blog post. These are generated by the AI at draft-confirm time and need to be persisted so the Publish step can read them without a second AI call.

## Options Considered

| Option | Pros | Cons |
|--------|------|------|
| Add columns to `blog_drafts` | Co-located with draft content, simple query, no join | Slightly wider table |
| New `blog_seo` table | Clean separation | Extra join, extra migration, overkill for two fields |

## Decision

Chosen: **Add columns to `blog_drafts`**, because the SEO fields are a direct property of the confirmed draft and are always fetched together with it. A separate table adds complexity with no benefit at this scale.

## Rollback Plan

1. Run `ALTER TABLE blog_drafts DROP COLUMN meta_description, DROP COLUMN suggested_slug;`
2. Verify with `SELECT column_name FROM information_schema.columns WHERE table_name = 'blog_drafts';`
3. No data loss — columns were nullable and newly added; no existing data references them

**Rollback tested against**: staging
**Rollback window**: any time before new-shape writes accumulate (safe indefinitely since columns are additive only)

## Cross-Service Consumers

- **blog-generator backend** — the only service reading/writing `blog_drafts`. No other consumers.
- **none** — no external ETL, analytics pipeline, or third-party integration reads this table

Deploy-order constraint: none — migration can run before or after backend deploy since new columns are nullable and old code ignores unknown columns.

## Testing Plan

- **Dev smoke**: run `npm run migrate` locally, then `SELECT meta_description, suggested_slug FROM blog_drafts LIMIT 1;` to confirm columns exist with NULL values
- **Staging verify**: apply migration, confirm via Supabase table editor that both columns appear; confirm existing draft rows are unaffected (NULL values, no errors)
- **Canary / phased rollout**: n/a — additive schema change, no risk window

## Observability

- **During apply**: watch Supabase logs for any lock errors (none expected for `ADD COLUMN` on a small table)
- **Post-apply**: confirm draft confirm API (`POST /api/blogs/:id/draft/confirm`) populates both fields; verify via `GET /api/blogs/:id/draft` returns `meta_description` and `suggested_slug`
- **Alerts armed**: none specific — monitor general API error rate in Supabase logs

## Consequences

- Enables Publish step to display and copy AI-generated meta description and slug without a second AI call
- Existing draft rows have NULL for both columns until re-confirmed (acceptable — Publish step renders empty state gracefully)
